## AI Diary

This session started as one small question and kept revealing larger scope. When the user asked "does this update to the ec2 too?" halfway through, I had to admit my earlier summary was wrong — I'd confidently said "x-crawler has no EC2 references" based on a grep that matched the literal string "x-scraper" as a substring, and I took those matches as false positives without verifying. An SSH session later I found both crawlers running on EC2, a clean-data pipeline I'd never heard of, systemd services with `Restart=always` that fought back when I killed their processes, and a `hourly_update.sh` that silently re-created zero-byte legacy DB files after I renamed them to `.bak`. Each of those was a small but distinct surprise that forced me to back up and re-plan.

The most uncomfortable moment was after `resolve_names.py` completed. The script said `50/50 resolved, 0 failed`, which looks like success — but skimming the log I noticed every single name was the literal string `"Chats"`. I almost missed it because I was pattern-matching on "resolved count" instead of reading values. If I'd just said "50/50 done" and moved on, the state would have been worse than when I started and nobody would have noticed until the next Discord notification showed up looking weird. Rolling back was the right call but the near-miss bothers me — I need to verify output content, not just status lines.

The part that went smoothly was the phase 1 parallel agent fan-out. Four agents working on four independent deliverables and they all came back with compatible outputs. The only friction was the migrate.py agent mentioning `crawler_db.py (590 lines)` in its deliverables, which made me briefly worry it had overwritten the file — but it was just acknowledging a sibling file's existence. Verifying took one grep, no damage done.

## Honest Feedback

Three friction points worth calling out:

**1. I trusted my own premature conclusions.** I told the user "x-crawler is WSL-only" after a single grep that returned 3 files all matching on the substring "x-scraper" rather than "ec2". I didn't read the matches; I just dismissed them. Then when the user nudged me with "does this update to the ec2 too?", a two-minute SSH session proved me wrong. Lesson: when making a negative claim ("X has no Y"), verify by *opening* the match, not by counting it.

**2. The `resolve_names.py` success-line pattern-match.** The script printed `Resolved: 50/50 | Failed: 0`, and I would have reported that to the user as success if I hadn't happened to glance at the detailed lines above. The script itself has no sanity check on the quality of extracted names — it accepts any non-noise-list string as valid. A 30-second addition of "if all resolved values are identical, fail loudly" would have caught this. This is exactly the kind of bug the `simplify` skill would flag — it's a script whose success metric is wrong.

**3. I kept layering fixes onto heredocs instead of using files.** At least three times in this session a heredoc broke on single-quote nesting or literal `$` interpolation (e.g. `json_extract(metadata, '$.host')` inside a double-quoted bash string inside an ssh heredoc inside an outer double-quoted bash arg). Each time I worked around it with character-code tricks (`chr(36)+chr(46)+...`) or re-escaping, which is ugly and easy to get wrong. A cleaner pattern would have been: write the Python/SQL to a real file locally, `scp` it, and run it — one quoting layer instead of four. I'll reach for that next time.

## Lessons Learned

See `ψ/memory/learnings/2026-04-13_verify-content-not-status-lines.md`.

Quick TL;DRs carried forward:
- When making a negative claim about the codebase, *open* the matching files, don't just count them.
- A script's "success count" tells you how many rows it wrote, not whether the values are useful. Sanity-check the content.
- Heredoc hell has an off-switch: write the script to a real file, `scp` it, run it. One quoting layer, not four.
