## Honest Feedback (3 friction points, 100+ words)

**1. I defaulted to "documented as limitation" instead of "let me try."** This is the big one. The retro from the main session blithely noted that Oracle Sync was blocked because `arra_learn` wasn't available, and I went on as if that were a fact of nature. It wasn't. It was a config gap I hadn't bothered to look for. This is the exact excuse pattern the anti-rationalization guard targets — "The API/tool didn't work" without showing what I tried. I didn't try anything. The fix cost me fifteen minutes including scope correction; the rationalization cost me nothing up front but left a real gap in the system.

**2. I added the MCP server at project scope first without considering why.** `claude mcp add` without `-s user` defaults to project-local scope, and I just used the default instead of asking "where do I actually want this server available?" It was only after the add succeeded that I realized a project-scoped Oracle memory layer makes no sense — the whole point is cross-project memory. Had to remove and re-add. Five-second mistake caught by five-second reflection, but I should have thought about scope before typing the command. Configuration defaults deserve the same sanity check as implementation defaults.

**3. The investigation path was inefficient.** I searched `~/.claude/`, `~/.openclaw/workspace/`, the `luna-oracle` project, and `maw.config.json` before finding `arra-oracle-v3` under `/mnt/c/Users/ttapk/ghq/`. The ghq root was literally mentioned in `maw.config.json` on the first file I read, but I skimmed past `"ghqRoot": "C:\\Users\\ttapk\\ghq"` without recognizing it as the key. A targeted grep for `arra` under `/mnt/c/Users/ttapk/ghq/` would have cut the search in half. Lesson: read config files for their pointers, not just their settings.

## Lessons Learned

See `ψ/memory/learnings/2026-04-13_mcp-servers-load-at-session-start.md` — captures two distinct insights:
1. "Blocked because tool missing" is almost always fixable — go find the tool before writing it off.
2. MCP servers register config immediately but don't expose tools until session restart. Adding one mid-session is a *next session* fix, not a this-session fix.

Also ties back to the earlier lesson from the 11.02 retro (`verify-content-not-status-lines.md`) — both are species of the same failure mode: accepting the surface of a situation (status line, missing tool) instead of checking what's actually underneath.
